Reverse Engineer Agent Harness Design

PAL — Harness Architecture & Evidence Report

Suy luận how PAL runs its agents under the hood — orchestration loop, tool interface, context/memory handling, guardrails, và human-in-the-loop — bằng cách quan sát behavior thực tế của một lần chạy (latency, errors, outputs, UI affordances) và đối chiếu với workflow configuration artifact. Mỗi hypothesis được gắn (tagged) trực tiếp với finding chứng minh nó. Evidence chỉ đến từ một run quan sát được — báo cáo phân biệt rõ điều gì là hypothesis chung cho nền tảng PAL và điều gì chỉ đúng cho run này (xem mục Limitations trong Conclusion).

4. Updated Conclusion

Phạm vi kết luận được giới hạn vào workflow configuration artifact và observed execution behavior của run đã ghi nhận.

Inferred Agent Harness Design — evidenced by one observed run

Orchestration loop: control-flow (Edges) và data-flow (typed reference) là hai cơ chế tách biệt (Finding 1, 2). Bên trong một node Agent, loop thực thi theo mẫu quan sát được: gọi tool → nhận lỗi dạng structured result (không fail) → agent tự điều chỉnh lời gọi → tiếp tục đến khi có output cuối (10, 11). Bên trong node Orchestrator-Worker, loop mở rộng thêm một bước phân rã-rồi-giao việc, với latency tổng (~140s) gợi ý nhiều khả năng là tuần tự hơn là song song (7).

Tool interface: mẫu hình discovery-rồi-invoke (search_workspace_tools → use_workspace_tool) với validation tại boundary và rate-limit được trả về dưới dạng structured, non-fatal result (10, 11, 12).

Context/memory handling: lắp ráp có chọn lọc theo template + reference đã resolve, không phải một transcript chia sẻ toàn cục; memory dài hạn được cấu hình tắt cho agent quan sát (2, 3, 13).

Guardrails: ít nhất 4 lớp riêng biệt được quan sát — schema validation, external rate limit pass-through, capability flags, và giới hạn số lượng subtask (10–14).

Human-in-the-loop: chỉ tại graph entry qua node Human Input chính thức; không có approval gate giữa các agent trong run này; các control sau khi chạy xong (Share/Export/New Run/Delete) là UI, không phải execution gate (4, 5).

Đây là các hypothesis về kiến trúc harness của PAL nói chung, được evidence hoá bằng behavior của một run cụ thể — không phải mô tả toàn bộ nền tảng đã được xác nhận. Phần nào là cơ chế (scheduler đọc Edges, resolver xử lý typed reference, tool gateway discovery-then-invoke) có khả năng tổng quát hoá cho các workflow khác; phần nào chỉ là giá trị cấu hình của riêng run này (6-node graph cụ thể, maxTaskCount=10, 2 worker agent cụ thể) thì không nên tổng quát hoá nếu chưa có thêm run khác để đối chiếu.

Còn lại Unknown: scheduler semantics chi tiết, hidden/shared context, durable persistence, retry policy tổng quát, worker concurrency thực tế (không có timestamp cấp task), capability enforcement, và hành vi khi vượt maxTaskCount.
